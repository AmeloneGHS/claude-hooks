# Features Research

**Domain:** CLI hook manager for Claude Code
**Researched:** 2026-03-18
**Confidence:** HIGH (official Claude Code hooks docs verified, competitor repos analyzed, git hook manager patterns studied)

## Competitive Landscape Context

Before categorizing features, here is what exists and where the gaps are:

| Tool | What It Is | Hooks | CLI | Testing | Registry |
|------|-----------|-------|-----|---------|----------|
| **disler/claude-code-hooks-mastery** (3.3K stars) | Script collection with 13 lifecycle hooks | Yes | No | No | No |
| **webdevtodayjason/claude-hooks** | CLI with 20+ hooks, enable/disable, doctor | Yes | Yes | Partial (`test` command) | Bundled |
| **carlrannaberg/claudekit** (633 stars) | Opinionated monolith, requires Max plan | Yes | Yes | No | No |
| **claude-hooks-sdk** (npm) | SDK for building hooks programmatically | No hooks | No | No | No |

**Key finding:** webdevtodayjason/claude-hooks is the closest competitor. It has install/enable/disable/create/doctor commands and 20+ hooks. Our differentiation must come from superior testing, packs/composition, npx-first DX, and bulletproof settings.json merge safety.

**Patterns borrowed from git hook managers:**

| Tool | Pattern We Adopt | Pattern We Skip |
|------|-----------------|-----------------|
| **Husky** | `.husky/` directory convention, shell script hooks, `npx` init workflow | package.json config (we use settings.json) |
| **Lefthook** | Parallel execution awareness, glob/regex matchers, local config overrides | YAML config (Claude Code uses JSON) |
| **pre-commit** | `run --all-files` testing, `try-repo` for external hooks, hook validation | Python ecosystem, remote hook repos in v1 |
| **lint-staged** | `--debug`, `--quiet`, `--dry-run` flags, concurrency control | File-scoped execution (our hooks are event-scoped) |

## Table Stakes

Features users expect. Missing any of these and users choose a competitor or just copy scripts manually.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| `init` command | Every CLI tool has setup. Creates `.claude/hooks/` dir, seeds settings.json hooks section | Low | Must detect existing settings.json and merge, never overwrite. Detect scope (user vs project) |
| `add <hook>` | Core value prop. Install a hook by name from bundled registry | Medium | Copies script, updates settings.json, sets executable bit (`chmod +x`). Atomic operation |
| `remove <hook>` | Users need clean uninstall | Medium | Remove script file AND settings.json entry. Must handle hooks referenced by packs |
| `list` command | Users need to see available hooks and which are installed | Low | Two views: installed vs available. Show event type, matcher, description. Color-code status |
| Non-destructive settings.json merge | Users already have hooks, permissions, MCP config. Overwriting = data loss = instant uninstall | High | JSON deep merge with conflict detection. **This is the #1 trust requirement.** Must preserve every non-hook field and existing hook entries not managed by us |
| Hook metadata/documentation | Users need to understand what a hook does before installing | Low | Each hook needs: name, description, event, matcher, what it blocks/allows, example input/output |
| Shell script hook format | Matches Claude Code's native format. Lowest barrier | Low | Bash scripts with proper stdin JSON parsing via `jq` or built-in |
| `npx` zero-install support | Modern CLI expectation. `npx claude-hooks init` must Just Work | Low | Package.json bin field, shebang, no postinstall side effects, fast startup |
| Exit code compliance | Claude Code uses exit 0 (allow), exit 2 (block), other (non-blocking error) | Low | All bundled hooks must follow this contract. Document it clearly |
| Cross-platform shell compatibility | macOS + Linux minimum. WSL for Windows | Low | POSIX-compatible bash. Avoid zsh-isms, macOS-only flags (`sed -i ''` vs `sed -i`) |
| Clear error messages | CLI tools with cryptic errors get abandoned fast | Low | Structured errors with what went wrong, why, and what to do. Use colors |
| `--help` on every command | Standard CLI UX | Free | Auto-generated by Commander.js |

## Differentiators

Features that set `claude-hooks` apart. Not expected, but create competitive advantage and drive word-of-mouth.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| **Hook testing framework** | No competitor does this well. Feed mock tool JSON to a hook, validate exit code + stdout JSON. `claude-hooks test <hook>` or `claude-hooks test --all` | Medium | Ship test fixtures per bundled hook. Users can test custom hooks too. **Single biggest differentiator.** Enables CI integration |
| **Hook packs/bundles** | `claude-hooks add security-pack` installs 3-4 related hooks at once. Competitor only has individual hooks | Low | Packs: `security-pack`, `quality-pack`, `cost-pack`. Packs are just named groups in registry metadata. One-command onboarding |
| **Hook scaffolding** | `claude-hooks create my-hook --event PreToolUse --matcher Bash` generates working hook with boilerplate, stdin JSON parsing, exit code handling | Medium | Template per event type. Include test fixture skeleton. Huge DX win for custom hooks |
| **Dry-run mode** | `claude-hooks add security-pack --dry-run` shows exactly what would change in settings.json without changing anything | Low | Builds trust. Users preview before committing. Critical for non-destructive reputation |
| **Hook conflict detection** | Warn when two hooks target the same event+matcher and might interfere | Low | Parse settings.json, find overlapping matchers, show precedence order |
| **`doctor` command** | Validate installation: scripts exist, are executable, settings.json valid, no broken references, correct shebangs | Low | Competitor has basic version. Ours checks: chmod, shebang, JSON schema, stdin parsing capability, jq availability |
| **`info <hook>` with examples** | Show what the hook blocks, what it allows, with real input/output JSON examples | Low | Embed examples in hook metadata. `claude-hooks info sensitive-path-guard` shows the exact JSON that triggers block vs allow |
| **Settings.json backup/restore** | `claude-hooks backup` before changes, `claude-hooks restore` if things break | Low | Copy to `.claude/settings.json.bak.{timestamp}`. Safety net for nervous users |
| **Scope awareness** | Install to user (`~/.claude/settings.json`), project (`.claude/settings.json`), or local (`.claude/settings.local.json`) | Medium | `--scope user\|project\|local` flag. Default to project. Mirrors Claude Code's own settings hierarchy |
| **Version pinning manifest** | Track which version of each hook is installed in `.claude-hooks.json` | Low | Enables `claude-hooks outdated` later. Know what's installed without parsing settings.json |
| **Hook validation on add** | Verify hook script is valid (executable, correct shebang, parses JSON input) before installing | Medium | Catch broken hooks before they block Claude Code workflows. Run a quick smoke test |

## Anti-Features

Features to deliberately NOT build. These seem appealing but create maintenance burden, complexity, or misaligned incentives.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| **Remote hook registry/marketplace** | Server infrastructure, moderation, security vetting, uptime responsibility. Massively increases scope for v1 | Bundle hooks with npm package. Community hooks via GitHub PRs. Add `--from <git-url>` in v2 if demand exists |
| **GUI/TUI interface** | CLI-first tool. TUI adds dependencies (ink, blessed), testing surface, accessibility concerns | Keep CLI clean. Users who want visual use `/hooks` in Claude Code itself (built-in read-only viewer) |
| **Auto-updating hooks** | Silent updates to security-critical hooks = trust violation. Users must control versions | Pin to package version. Users update via `npm update claude-hooks` |
| **Hook auto-discovery from filesystem** | Scanning for scripts and auto-registering creates security concerns and confusion about what's active | Explicit `add` only. User must opt-in to each hook |
| **Prompt/agent hook scaffolding by default** | These cost API tokens (prompt hooks call Haiku, agent hooks spawn subagents). Shipping hooks that cost users money without clear warning is hostile | Default to `command` type hooks only. Document prompt/agent types but don't scaffold them. If we ever do, show estimated cost per invocation |
| **Windows native (.bat/.ps1) support** | Claude Code is macOS/Linux focused. Shell hooks are bash. Maintaining Windows variants doubles maintenance | Document WSL as the path for Windows. Ship bash scripts only |
| **Organization/managed policy hooks** | Enterprise feature requiring admin infrastructure, auth, policy enforcement | Out of scope. Anthropic handles this natively via managed settings |
| **Hook analytics/telemetry** | Privacy concern, community trust killer for open-source tool | No phone-home ever. Local-only `doctor` diagnostics |
| **MCP server for hook management** | Adding an MCP server overcomplicates what should be a CLI tool. Running services != CLI tool | Stay CLI-only. Hooks are files + JSON config, not running services |
| **Claude Code API/process integration** | No public API exists. Injecting into Claude Code process is fragile and unsupported | Read/write settings.json and hook script files only. File-based interface |
| **Hook chaining/dependency resolution** | Complex DAG resolution between hooks. Over-engineering for event-based system | Each hook is independent. Users compose via settings.json array ordering (first block wins) |
| **Interactive TUI menus** | Overhead, dependency bloat, not needed for <10 commands | Clean non-interactive CLI with `--help`. Competitor has interactive menu; we beat them with better DX, not more UI |

## Feature Dependencies

```
init ──────────────────────────┐
                               v
                 settings.json merge engine (CORE)
                               │
              ┌────────────────┼────────────────┐
              v                v                v
            add             remove            list
              │                │
              v                v
        hook registry    settings.json cleanup
              │
              ├──────────────────────┐
              v                      v
        individual hooks       hook packs (groups)
              │
              v
        hook metadata ──────────> info command
              │
              v
        test fixtures ──────────> test command
                                    │
                                    v
                               dry run mode

create (scaffolding) ──> generates: hook script + metadata + test fixture

doctor ──> validates: file perms, settings.json schema, script existence, shebang

backup/restore ──> wraps: settings.json snapshot with timestamp
```

**Critical path:** `init` -> `settings.json merge engine` -> `add/remove` -> `list`. Everything else layers on top.

The merge engine is the foundation. If settings.json management is unsafe, nothing else matters — users will uninstall immediately.

## MVP Recommendation

### Phase 1: Core CLI (must ship for launch)

Prioritize:
1. **`init`** — Foundation for everything else
2. **Settings.json merge engine** — Non-destructive, conflict-aware (THE trust builder)
3. **`add <hook>` / `remove <hook>`** — Core value loop
4. **`list`** — Discovery and status
5. **5-8 bundled hooks** — Security (sensitive-path-guard, command-blocker), Quality (post-edit-lint), Cost (web-budget-gate), Error (error-advisor)
6. **Hook metadata** — Name, description, event, matcher per hook
7. **Hook packs** — `security-pack` is the hero demo for launch (`npx claude-hooks init && npx claude-hooks add security-pack`)

### Phase 2: Testing + DX (the differentiator release)

1. **`test <hook>`** — Run hook with mock input, validate output. THE differentiator
2. **`test --all`** — Run all installed hook tests
3. **`create <hook>`** — Scaffold new hook with boilerplate + test fixture
4. **`doctor`** — Validate installation health
5. **`info <hook>`** — Detailed hook docs with input/output examples
6. **`--dry-run` flag** — Preview settings.json changes before applying

### Phase 3: Polish (retention + trust)

1. **`backup` / `restore`** — Settings.json safety net
2. **`--scope` flag** — user/project/local installation targets
3. **Conflict detection** — Warn on overlapping matchers
4. **Version pinning manifest** — `.claude-hooks.json` tracking
5. **Hook validation on add** — Smoke test before install

**Defer indefinitely:** Remote marketplace, GUI/TUI, auto-update, telemetry, Windows native, MCP server, prompt/agent hook scaffolding

## Complexity Budget

| Complexity | Count | Features |
|------------|-------|----------|
| Free | 1 | --help (Commander.js) |
| Low | 12 | init, list, metadata, exit codes, shell compat, npx, packs, dry-run, conflict detection, doctor, info, backup |
| Medium | 5 | add, remove, merge engine, test framework, scaffolding |
| High | 0 | None — deliberately avoiding high-complexity features in v1 |

**Total estimated effort:** Medium. The merge engine and test framework are the only non-trivial pieces. Everything else is standard CLI tooling patterns well-established by Husky, Lefthook, and pre-commit.
